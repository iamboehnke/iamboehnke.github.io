What it does

Every morning at 06:00, Daglig Vejr pulls weather and pollen data from two public sources, runs the numbers through a classification layer calibrated to my personal allergy profile, and sends me an HTML email with concrete recommendations for the day: what to wear, whether to apply sunscreen, whether to take a hay fever pill, and whether to bring an umbrella.

I have a class 6 grass pollen allergy (IgE 189 kUA/L - the highest measurable class) and I run outside most mornings, so getting these things wrong in combination is genuinely unpleasant. The project exists because I wanted something that actually runs, produces a tangible output I use every day, and demonstrates end-to-end production thinking rather than living in a notebook.

Data sources

Weather data comes from Open-Meteo, a free meteorological API requiring no key or registration. It provides all variables I need for Odense: UV index, precipitation probability, cloud cover, wind speed, apparent temperature, and daily forecasts. UV index is particularly important for the sunscreen recommendation and is not available on DMI's free observation tier.

Pollen measurements come from Astma-Allergi Danmark via their internal JSON API endpoint, which is the same data source used by their official Dagens Pollental app. The endpoint serves a Firestore-format document with measurements from two Danish stations; I use station 48 (Copenhagen/East Denmark), which is most representative for Odense and Funen.

Why not DMI for pollen? DMI does not expose a public pollen API. Astma-Allergi Danmark's endpoint is undocumented but has been stable and is used by multiple independent integrations. The fetcher fails gracefully to zero values if the endpoint is unreachable, which is the appropriate behaviour outside the April–September pollen season.

Personalised thresholds

Standard pollen advisory systems recommend an antihistamine when grass pollen exceeds 30 grains/m³ - the population-level "moderate" threshold. With a class 6 IgE, that number is far too high. My specific IgE results from a hospital test informed the thresholds used by the system:

Grass pollen (IgE 189 kUA/L, class 6): pill from 5 grains/m³ Birch pollen (IgE 55 kUA/L, class 5): pill from 15 grains/m³ Mugwort (IgE 3 kUA/L, class 2): pill from 30 grains/m³

These thresholds are the starting point, not the endpoint. The feedback loop described below is designed to refine them further based on my actual day-to-day experience.

Architecture

The system has three jobs, each a standalone Python script triggered by a GitHub Actions workflow. No server, no container, no persistent process - just scripts that run on schedule and exit.

Open-Meteo API ──┐ ├──► weather_job.py ──► rules.py ──► email_sender.py ──► Gmail Astma-Allergi ──┘ │ └──► data/history.json (append-only, committed to repo) │ feedback_job.py ─┘ (triggered by GitHub Issues) │ train_job.py ────┘ (weekly, Saturday)

Daily job

weather_job.py runs at 04:00 UTC (06:00 CET) via a GitHub Actions cron schedule. It fetches both data sources, loads any trained ML model, applies the classification rules with optional model-derived threshold adjustments, sends the email, and appends the day's record to data/history.json. The JSON file is committed back to the repository at the end of each run, making the history visible in the commit graph.

Feedback loop

Each email contains two links: "Ja, det var præcist" and "Nej, passede ikke". Clicking one opens a pre-filled GitHub Issue. A second workflow, triggered on issue creation, parses the issue title, writes the feedback into history.json, and closes the issue automatically. The whole interaction - from clicking a link in the email to the feedback being stored - takes under thirty seconds and requires no manual steps beyond the click.

ML layer

A third workflow runs every Saturday. Once 20 or more labelled daily records have accumulated, it trains a Random Forest classifier on 13 features (temperature, UV index, pollen counts, season, day of week, and others) with the binary feedback label as the target. The trained model does not replace the rule-based logic; it produces threshold adjustments - lowering the pollen pill threshold when the model predicts the rules are being too conservative, for instance - that the rules engine applies on top of its defaults. Cross-validation accuracy and feature importances are saved to data/model_metrics.json and committed alongside the model.

The separation between rules and model is a deliberate design choice. Rules produce explainable outputs from day one. The model improves personalisation over time without making the system opaque or dependent on sufficient training data to function at all.

Deployment

Everything runs on GitHub Actions using the free tier for public repositories. The three workflows together use well under one minute of compute per day, a negligible fraction of the monthly free allowance. There is no infrastructure to maintain, no server to keep running, and no cost.

Total monthly cost: zero. Open-Meteo is free, Astma-Allergi's data is free for personal use, Gmail app passwords are free, and GitHub Actions free minutes cover this workload with several orders of magnitude to spare.

What I learned

The first lesson was about API surface area. Open-Meteo is clean and reliable. The Astma-Allergi endpoint returns a Firestore-format document, which requires navigating a deeply nested key structure and handling the difference between integerValue and doubleValue types. More interestingly, the API returns -1 for species with no measurement on a given day rather than omitting the key. That kind of detail does not appear in any documentation; it only becomes visible when you look at actual responses across different dates and species.

The second lesson was about calibration. Writing a system that sends a recommendation you act on every morning makes correctness feel concrete in a way that a held-out test set does not. Getting an antihistamin recommendation on a zero-pollen day because a threshold is set too low is immediately noticeable. The feedback mechanism exists precisely to close that loop systematically rather than adjusting thresholds by hand when something feels off.

The third was about production discipline. It is easy to write code that works once on a laptop. It is harder to write code that runs at 06:00 every morning, commits its outputs to version control, processes asynchronous user input via GitHub Issues, and recovers cleanly when an upstream API returns unexpected data. This project forced me to think about observability and failure modes in a way that notebook-based work does not.
